# プロジェクト名称: Kakeibo App 構造改革 (Refactoring & Robustness)

## 概要
肥大化した JavaScript ファイルを機能単位（コンポーネント）に分割し、AI が特定の機能を修正しても他の機能（例: ピン留め）が消えない堅牢な構造を構築する。

## コア機能一覧

| # | 機能名 | 説明 | 優先度 |
|---|--------|------|--------|
| 1 | スクリーン分割 | analysis.js や input.js 等の巨大ファイルを UI コンポーネントに分割 | Must |
| 2 | ストアのモジュール化 | store.js を状態（State）ごとに細分化し、副作用を限定する | Must |
| 3 | クリティカル機能の保護 | ピン留め、口座順序変更などのロジックを独立したモジュールに抽出 | Must |
| 4 | 自動回帰テストの整備 | Evaluator が「ピン留めが機能しているか」を常にチェックする体制を構築 | Should |

## スプリント計画

### Sprint 1: 共通基盤のモジュール化
**ゴール:** 巨大な `store.js` を分割し、ES Modules への移行を完了する。
**受け入れ基準:**
- アプリが正常に起動し、データの読み書きがこれまで通り行えること。
- Store の各機能（口座、履歴、設定）が個別のファイルに分離されていること。

### Sprint 2: スクリーンコンポーネントの解体（分析画面）
**ゴール:** 最も巨大な `analysis.js` を「グラフ」「集計」「口座一覧」などの部品（コンポーネント）に分ける。
**受け入れ基準:**
- `analysis.js` が 100行程度の「呼び出し専用」ファイルになること。
- ピン留め機能と口座並び替え機能が独立し、互いに干渉しないこと。

### Sprint 3: 回帰テストスイートの構築
**ゴール:** Evaluator が自動で主要機能を巡回テストできるシナリオを作成する。
**受け入れ基準:**
- `/evaluator` を実行した際、「ピン留め表示チェック」「口座名変更テスト」が自動で走り、結果が報告されること。

## UI/UX 要件
- 見た目の変更は行わず、コード構造のみを整理する（リファクタリング）。

## 制約事項
- Vanilla JS（素のJavaScript）と ES Modules のみを使用し、外部フレームワークは導入しない。
