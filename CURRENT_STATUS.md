# Kakeibo Sync Engine 現状分析レポート (2026-05-13)

## 1. 完了した修正 (v7.5)

### 同期ロジックの安定化
- **「保留キュー (Pending Queue)」の導入**:
    - 高速な連続操作（追加・削除の連打）時に、同期リクエストが無視される問題を解決。
    - 通信中に次の操作が発生した場合、キューに溜めておき、完了直後に自動で再実行する。
- **読み込み専用モード (pullOnlyFromCloud) の新設**:
    - WebSocket通知による他デバイスからの更新時、無駄な「読み書き往復」を廃止。
    - API呼び出し回数を半減させ、レートリミットや競合エラーを抑制。
- **マージ優先度の最適化**:
    - `save()` 時はローカル優先、`pullFromCloud()` 時はクラウド優先を明示的に強制（forcePriority）。
    - これにより「削除したはずのデータが他デバイスから復活する」バグを根絶。

### UI/UX
- **編集モーダルのアイコン整理**:
    - `<select>` タグ内で Lucide アイコン（HTML）が使えない制限に対し、名前のみの表示に統一してシンプル化。
- **アイコンマッピングの補完**:
    - `📁` や `💱` など、旧バージョンの絵文字アイコンが履歴一覧で `❓` になる問題を、Lucide へのマッピング追加で修正。

## 2. 現在の既知の課題 (Potential Bugs)

### クラウド側の整合性
- **アトミック性の不完全さ**:
    - 現在、`batchUpdateValues`（書き込み）と `batchClear`（余剰行削除）を別々に実行している。
    - まれに、書き込み成功・クリア失敗が起きると、クラウド上に古い重複行が残る可能性がある。
    - **対策案**: Google Sheets API の `requests` パラメータを使用して、1回のリクエストで更新と削除を同時に行うように統合。

### 同期タイミング
- **localStorage との状態同期**:
    - `SyncManager` が直接 `localStorage` を操作している部分があり、UIの状態管理（Store）との整合性が極めて高い頻度の操作でズレる可能性がゼロではない。

## 3. 次のステップ

1. **Google Sheets API の 1-Request 統合**: 同期エンジンのアトミック性を100%にする。
2. **エラーハンドリングの強化**: 通信エラーが連続した場合の指数バックオフ（リトライ間隔の調整）の導入。
3. **データ圧縮**: 送信データ量が多い場合の最適化。
